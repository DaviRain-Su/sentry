# Sentry — Build Status

日期：2026-06-02 · 环境：Sui Testnet

A running snapshot of what is built and how it was verified. Demo-facing summary for judges / handoff.

## Product direction update (2026-06-03)

Production direction is now **Agent dispatch platform**. The Sentry daemon manages policies, Guardian checks, and dispatches tasks to external Agents (Claude Code / Codex / Kimi). The daemon does NOT execute trades itself — external Agents use local toolchains (OWS, Solana CLI, wallets) for signing and execution.

Current production target slice is **Solana + Ethereum + Hyperliquid + OKX**. Sui Testnet remains the verified demo path.

> ⚠️ **现实校准**：上述方向目前主要落在文档层，代码主体仍是上一版云端自执行 Agent。
> 文档愿景 vs 代码现实的逐项对照、矛盾点与复核方法见 [`docs/12-reality-vs-vision.md`](12-reality-vs-vision.md)。

- `core/ethereum-trade.js` now builds and validates Ethereum swap `submit_tx` AgentTasks for external Agents, rejects withdrawal scope, uses quote id as idempotency evidence, and lets the dispatcher verify EVM transaction hash / quote id / venue / chain evidence under a task-local daemon readiness override when env wallet proof matches.
- `agent/src/local-signer-probe.mjs` now adds a non-signing Solana/Ethereum signer-address probe from explicit signer env or local probe command, exposed through `sentry-daemon signer probe` and remote `signer.probe`.
- `agent/src/solana-receipt-adapter.mjs` and `agent/src/ethereum-receipt-adapter.mjs` now add mock-tested RPC receipt polling for submitted/done chain tasks: Solana `getSignatureStatuses` and Ethereum `eth_getTransactionReceipt`, wired into daemon `agent.dispatch` receipt verification.
- `agent/src/local-activity-log.mjs` now writes sanitized local daemon activity JSONL for blocked and completed `agent.dispatch` attempts, exposes `sentry-daemon activity tail`, and enables remote read-only `activity.tail` without returning raw secret-shaped fields.

- `agent/src/local-policy-task-planner.mjs` now turns due local policies with explicit `task_template(s)` / `planned_tasks` into planned AgentTasks for OKX, Hyperliquid, Solana and Ethereum through `sentry-daemon policy plan` and remote `policy.local.plan`; it remains pre-dispatch and does not run Guardian automatically.
- `agent/src/local-policy-runner.mjs` now adds `sentry-daemon policy run-once` and remote `policy.local.run_once`: due tasks pass through local policy guard, optional readiness checks and optional registered-Agent dispatch. Dispatch is off unless explicit.
- `agent/src/local-policy-loop.mjs` now adds the daemon-owned periodic policy loop controller. It supports `--policy-loop`, remote `policy.local.loop.status/start/stop/run_now`, overlap protection, last-run summaries and no-dispatch defaults.
- `agent/src/local-agent-capability-probe.mjs` now adds `sentry-daemon agent probe` and remote `agent.probe`: registered external Agent commands are probed with bounded `--version`, declared baseline capabilities are checked, and secret-shaped output is redacted.
- Profile / Wallet now exposes a daemon control plane: OWS vault, Worker bridge, registered Agents, venue accounts and asset sources.
- Profile / Wallet now includes a target integration matrix for Solana, Ethereum, Hyperliquid and OKX, including authorization model, budget guard and next adapter step.
- New Strategy defaults to Local (daemon + Agent dispatch) and labels Remote Worker as the optional Sui Testnet path.

## Known gaps / next

The repo now has local registered Agent metadata, basic command/version/capability probes, local policy metadata/due tick state, due-policy AgentTask planning, one-shot guarded run-once dispatch, a daemon-owned periodic policy loop controller, a first AgentTask/AgentTaskResult stdio round trip and sanitized dispatch activity JSONL, but does not yet: (a) remove self-execution ExecutorAdapter code from Worker demo runtime, (b) run target venue receipt verification against real accounts, (c) implement deep task-level capability discovery for Claude Code/Codex/Kimi, (d) complete production Guardian/inventory TickLoop semantics.
1. **Target venue adapters** — Solana, Ethereum, Hyperliquid and OKX are now represented in shared catalog, authorization registry and inventory source registry. Read-only live inventory adapters exist under mock-fetch tests. OKX has a `place_order` AgentTask/result verifier, local dispatch-ready gate, order-status adapter and daemon receipt verifier skeleton. Hyperliquid has a `place_order` AgentTask/result verifier, local dispatch-ready gate, local agent-wallet grant metadata gate, public `userRole` live grant checker, pre-signed exchange-submit adapter, local nonce store, public order-status adapter and daemon receipt verifier skeleton. Solana and Ethereum have swap `submit_tx` AgentTask/result verifier, env-account local dispatch-ready gates, non-signing signer/address probes and RPC receipt polling skeletons. Real Solana/Ethereum signed trade execution adapters and Hyperliquid live-account dry-runs/UI wiring are still missing.
2. **AuthorizationAdapter implementation** — the shared registry and preflight validator now exist, but per-venue grant/revoke/read-state implementations are still missing for Solana, Ethereum, Hyperliquid and OKX.
2a. **SecretStore implementation** — metadata-only key handles can now be added/listed/removed through `sentry-daemon venue ...` and loaded by daemon `secret.store` / `inventory.sync`; OKX macOS Keychain status/store/read exists. OKX dispatch receipt verification now requires local operational proof (`read`, `place_order`, no `withdraw`, `ip_allowlist=true`). Still missing: cross-platform keyring, venue-side proof hardening and rotation.
2b. **OKX live read + trade/status enablement** — explicit `inventory.sync live=true` can now resolve local `SENTRY_OKX_*` env or macOS Keychain credentials and perform a read-only balance fetch under test. `core/okx-trade.js` can build/verify a spot `place_order` AgentTask, `agent/src/local-dispatch-readiness.mjs` can locally approve OKX dispatch after metadata/IP/credential proof, `agent/src/okx-order-status-adapter.mjs` can fetch status by `ordId` / `clOrdId`, daemon `agent.dispatch` now has receipt verification, and OKX fetches use bounded retry/backoff under tests. Still missing: production UI wiring and stronger venue-side permission proof.
2c. **Hyperliquid live read + trade/status enablement** — explicit `inventory.sync live=true` can now query Hyperliquid public account state under test when `read_account_address` or `SENTRY_HYPERLIQUID_USER_ADDRESS` is configured, with bounded retry/backoff. `core/hyperliquid-trade.js` can now build/verify a perps `place_order` AgentTask and dispatcher result evidence; daemon local readiness can approve a task-local Hyperliquid dispatch override after metadata/read-address proof and active agent-wallet grant metadata; daemon `agent.dispatch` defaults to checking public `info.userRole` for the agent wallet before Hyperliquid dispatch and receipt verification; daemon can submit an external Agent's pre-signed `/exchange` payload, persist a local `authorization_ref + nonce` claim by default to block duplicate signed-submit reuse, and then query public `orderStatus` to enrich accepted dispatch results. Still missing: production UI wiring, live-account dry-runs and end-to-end live submit testing.
2d. **Solana/Ethereum live read + swap-task enablement** — explicit `inventory.sync live=true` can now query local Solana/Ethereum RPC endpoints under test when wallet address env is configured. Solana and Ethereum now also have swap `submit_tx` AgentTask/result verifier skeletons, env-account task-local dispatch-ready gates, non-signing signer/address probes, and daemon RPC receipt polling (`getSignatureStatuses`, `eth_getTransactionReceipt`) under mock-fetch tests. Still missing: OWS account discovery, token metadata registry, rate-limit/backoff policy, production UI wiring, Solana transaction build/simulation/signing handoff and Ethereum Safe/session-key grant installation, account discovery, real signature probing, calldata construction, simulation/signing handoff.
3. **DBUSDC funding** — the only true remaining Sui demo execution gap, and **self-funding is confirmed impossible** on this testnet: DBUSDC `mint` is TreasuryCap-gated (cap not public), DEEP `mint` returns `FunctionNotFound` on the current package, and a SUI→DBUSDC swap needs DEEP for taker fees (a zero-DEEP swap fills 0 even with a live bid). Needs an **external DBUSDC source** (DeepBook-team faucet, or an address that already holds DBUSDC/DEEP). Once the agent BalanceManager holds DBUSDC: flip `EXECUTION_ENABLED=true` and replay the execution PTB (`worker/src/deepbook.js` `buildExecutionTx`).
4. **Browser wallet click-through** — connect Slush (testnet) and run create/revoke from the UI against `VITE_WORKER_URL` (the on-chain txs above prove the underlying Worker path; the current UI now uses Worker-built txs again).
5. **zkLogin live test** (optional) — only if using Enoki instead of a wallet.
