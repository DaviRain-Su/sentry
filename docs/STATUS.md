# Sentry — Build Status

日期：2026-06-02 · 环境：Sui Testnet

A running snapshot of what is built and how it was verified. Demo-facing summary for judges / handoff.

## Product direction update (2026-06-03)

Production direction is now **Agent dispatch platform**. The Sentry daemon manages policies, Guardian checks, and dispatches tasks to external Agents (Claude Code / Codex / Kimi). The daemon does NOT execute trades itself — external Agents use local toolchains (OWS, Solana CLI, wallets) for signing and execution.

Current production target slice is **Solana + Ethereum + Hyperliquid + OKX**. Sui Testnet remains the verified demo path.

> ⚠️ **现实校准**：上述方向目前主要落在文档层，代码主体仍是上一版云端自执行 Agent。
> 文档愿景 vs 代码现实的逐项对照、矛盾点与复核方法见 [`docs/12-reality-vs-vision.md`](12-reality-vs-vision.md)。

- `agent/src/local-policy-store.mjs` now persists local policy metadata to `~/.sentry/policies.json`, exposes `sentry-daemon policy add/list/pause/resume/revoke/tick`, computes due-policy ticks, and enables remote metadata/control commands `policy.local.add`, `policy.local.list`, `policy.local.tick`, `policy.pause`, `policy.resume` and `policy.revoke`. - `agent/src/local-policy-task-planner.mjs` now turns due local policies with explicit `task_template(s)` / `planned_tasks` into planned AgentTasks for OKX, Hyperliquid, Solana and Ethereum through `sentry-daemon policy plan` and remote `policy.local.plan`; it remains pre-dispatch and does not run Guardian automatically.
- `agent/src/local-policy-risk-guard.mjs` now adds opt-in local inventory/risk checks for policy ticks: configured policies can require fresh inventory snapshots, minimum available balances, max per-asset position value and projected venue exposure caps before readiness or dispatch.
- `agent/src/local-policy-trigger-guard.mjs` now adds explicit local market trigger checks for due policy tasks. It supports price above/below, price drop/rise bps, funding-rate and venue-health triggers against a supplied `market_snapshot` / `--market-snapshot`; missing trigger data blocks readiness/dispatch, and unsatisfied triggers return no-op/skipped without spawning an external Agent.
- `agent/src/local-market-snapshot.mjs` now builds one-shot public market snapshots for trigger checks from OKX public ticker/funding endpoints and Hyperliquid public `allMids` / `metaAndAssetCtxs`; `sentry-daemon market snapshot`, `policy run-once --live-market` and remote `live_market=true` can use this without API keys.
- `agent/src/local-policy-runner.mjs` now adds `sentry-daemon policy run-once` and remote `policy.local.run_once`: due tasks pass through local policy guard, market trigger guard, optional inventory/risk guard, optional readiness checks and optional registered-Agent dispatch. Dispatch is off unless explicit.
- `agent/src/local-policy-loop.mjs` now adds the daemon-owned periodic policy loop controller. It supports `--policy-loop`, remote `policy.local.loop.status/start/stop/run_now`, overlap protection, optional market snapshots, optional inventory checks, last-run summaries and no-dispatch defaults.
- `agent/src/local-agent-capability-probe.mjs` now adds `sentry-daemon agent probe` and remote `agent.probe`: registered external Agent commands are probed with bounded `--version`, declared baseline capabilities and task-level venue/action capabilities are reported, and secret-shaped output is redacted.
- `agent/src/daemon-bridge-lifecycle.mjs` now makes Worker bridge revoke terminal for the daemon connection: `session_revoked` / close code `1008` stops heartbeat/reconnect, stops the remote-started process and policy loop, and prevents retrying with an invalidated relay token.
- `agent/src/daemon-identity-store.mjs`, `core/daemon-identity.js` and `worker/src/daemon-identity.ts` now add local Ed25519 daemon identity for pairing proof, relay-token refresh proof and daemon-origin WebSocket envelope proof. The daemon creates/loads `~/.sentry/identity.json`, signs the canonical pairing payload, and Worker verifies the proof before consuming the pairing code or issuing a relay token. For refresh, the daemon requests a Worker challenge, signs it with the same local identity, and Worker verifies the stored public key before rotating the short-lived relay token.
- `core/bridge-envelope.js`, `agent/src/bridge-envelope.mjs` and `worker/src/worker-bridge-identity.ts` now add bidirectional relay-token-bound HMAC signed envelopes plus bidirectional Ed25519 bridge identity signatures, relay-session monotonic `seq` and timestamp checks for Worker bridge messages. The daemon sends the relay token through WebSocket subprotocol by default; daemon messages (`hello`, `heartbeat`, `command_result`) carry both HMAC and paired Ed25519 `agent_signature`. AgentSession also persists a Worker bridge Ed25519 identity, sends its public key metadata in signed `session_accepted`, and signs Worker messages (`session_accepted`, `session_revoked`, `command`, `error`) with `worker_signature`. Messages are rejected on the receiving side if unsigned, tampered, stale/far-future by `issued_at`, missing `seq` or replayed for the same relay session; AgentSession rejects daemon-origin messages missing or failing the paired Ed25519 signature, and daemon rejects Worker-origin messages missing or failing the accepted Worker signature. Worker commands must include `expires_at`; the daemon blocks expired commands before dispatch and returns an error result for Dashboard polling. Worker AgentSession stores sequence state in Durable Object storage, and `agent/src/bridge-sequence-store.mjs` persists daemon sequence state to `~/.sentry/bridge-sequences.json` under a derived relay-token key, not the raw relay token or HMAC key. Relay-token challenge refresh now rotates token hashes, resets token-scoped sequence counters, closes the old WebSocket and lets the daemon reconnect with the new token. AgentSession also keeps a bounded pending queue for low-risk read/status commands while the daemon is offline, expires stale pending commands and replays active ones on reconnect; `POST /commands` deduplicates caller-provided `idempotency_key` and rejects type conflicts. Command records now track `queued` -> `acknowledged` -> `result`, store `command_ack`, finalize sent unacked commands as `COMMAND_ACK_TIMEOUT`, and send internal `command.resume` probes for acknowledged commands after reconnect. The daemon persists sanitized bounded command results in `~/.sentry/command-results.json`, returns stored results when available, reports `pending` for same-process in-flight commands, or finalizes the original command as `COMMAND_RESUME_NOT_FOUND`. High-risk dispatch/control commands still fail fast while offline and are never replayed across reconnect.
- Profile / Wallet now exposes a daemon control plane: OWS vault, Worker bridge, registered Agents, venue accounts and asset sources. The Worker bridge panel can create a pairing code, show the daemon CLI command, list indexed AgentSessions, read AgentSession status, submit remote `agent.probe`, queue `activity.tail`, fetch read-only `wallet.refs`, request metadata-only `authorization.state`, render the returned `readiness_summary` by blocked/planned/metadata-ready/dispatch-ready target venue, render venue-key rotation state, seed a local Hyperliquid/OKX policy through `policy.local.add`, request `policy.local.list` and render the sanitized local policy-store summary, run `policy.local.run_once` plan/preflight commands with `dispatch=false`, explicitly arm one-shot dispatch or dispatch-mode loop runs, require signer probe evidence, set signer handoff timeout, toggle public OKX/Hyperliquid market snapshots and metadata/live inventory preflight, control `policy.local.loop.status/start/run_now/stop`, run confirm-gated local `authorization.rotate`, run confirm-gated local `authorization.revoke`, revoke the bridge, and poll bounded command records by command id. The `authorization.rotate` and `authorization.revoke` UI controls are deliberately local-only: rotate records local proof after venue key material was rotated elsewhere; revoke disables local metadata/signing capability and refreshes `authorization.state`; both still report live venue/chain changes as pending manual proof.
- Worker AgentSession command records now store safe command payload summaries plus recursively sanitized daemon result payloads. Result storage redacts secret-shaped fields such as API keys, tokens, passphrases, mnemonics and private keys, while preserving non-secret execution evidence such as transaction hashes, signatures and venue order ids. Daemon-side `agent/src/local-command-result-store.mjs` applies the same bounded/redacted storage rule for local resume lookup.
- `npm run local-bridge:smoke` now gives a repeatable local Worker + daemon bridge smoke: it starts Wrangler and the daemon with a temporary state directory, runs `POST /api/local-agents/pairing`, waits for the daemon session to become online, loads metadata-only OKX/Hyperliquid venue handles plus metadata-only OWS Solana/Ethereum wallet refs, verifies remote `authorization.state` returns OKX/Hyperliquid metadata-ready and Solana/Ethereum planned grant gaps, verifies online-only `authorization.rotate` can locally mark Hyperliquid key metadata rotated and make `authorization.state` report `rotation_state=fresh`, sends remote `policy.local.add` for Hyperliquid/OKX and Solana/Ethereum policies, verifies `policy.local.list`, verifies `policy.local.plan` produces four planned tasks, runs no-dispatch `policy.local.run_once` readiness with a market snapshot and confirms four ready tasks and zero dispatched tasks, verifies online-only `authorization.revoke` can locally revoke OKX key metadata plus OWS wallet refs, verifies post-revoke `authorization.state` becomes blocked, and verifies a later plan/readiness run is reduced to one Hyperliquid-ready task plus three locally blocked OKX/Solana/Ethereum tasks, exercises relay-token challenge refresh with the temporary daemon identity, stops the old daemon, queues a low-risk `activity.tail` command while offline, verifies duplicate `idempotency_key` submit returns the same command record and a type conflict is rejected, restarts daemon with the refreshed relay token and verifies the offline command replays to a result. The smoke policies are generated by the same `src/local-policy-metadata.js` builder as New Strategy Local Agent deploy, and the temporary policy/venue/wallet/activity/command-result files must not contain raw secret fields or smoke secret material.
- New Strategy Local Agent deploy now builds sanitized local policy metadata through `src/local-policy-metadata.js` from the reviewed venue strategy and registers it through remote `policy.local.add` when a paired daemon owner-control token exists. Before registration it refreshes daemon `authorization.state` and required `wallet.refs`, injects real OKX/Hyperliquid key refs plus Solana/Ethereum OWS account refs into task templates, and blocks deploy when those refs are missing or locally blocked instead of persisting placeholder accounts. The UI metadata builder is test-covered for OKX, Solana, Ethereum and Hyperliquid task templates, and the deploy path no longer describes the local template requirement as OKX/Hyperliquid-only. Without Worker bridge configuration it shows a blocked state instead of pretending the policy was deployed.
- Profile / Wallet now includes a target integration matrix for Solana, Ethereum, Hyperliquid and OKX, including authorization model, budget guard and next adapter step.
- New Strategy defaults to Local (daemon + Agent dispatch), labels Remote Worker as the optional Sui Testnet path, and routes venue-scoped Local deploys to daemon policy metadata registration instead of the legacy Sui wallet-signing path.
